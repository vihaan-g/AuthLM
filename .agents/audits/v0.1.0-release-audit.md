## Bugs

| # | Severity | Description | Evidence | Impact | Suggested Fix |
|---|---|---|---|---|---|
| 1 | **HIGH** | `httpx.AsyncClient` leaked on every `connection_methods()` call | `providers/openai.py:90`, `anthropic.py:118`, `google.py:67`: `client = self._http_client or httpx.AsyncClient()` — never closed | Steady connection leak; CLI bounded by process lifetime but library usage leaks indefinitely | Inject shared client or close after connect |
| 2 | **HIGH** | Missing HTTP timeout on device-code poll POST | `oauth_device.py:184-191`: no `timeout=` parameter | If token endpoint hangs, request blocks for OS TCP default timeout (~2 min) per iteration | Add `timeout=30.0` |
| 3 | **HIGH** | Missing HTTP timeout on PKCE token exchange POST | `oauth_pkce.py:244`: no `timeout=` parameter | Same as above | Add `timeout=30.0` |
| 4 | **MEDIUM** | PKCE callback authorization error silently swallowed | `oauth_pkce.py:155-162`: `raise TokenEndpointError` in daemon thread handler — caught by `BaseHTTPRequestHandler` and lost | User waits full 300s timeout instead of getting immediate error | Set a captured error field and check it in `_wait_for_code()` |
| 5 | **MEDIUM** | `refresh()` succeeds at token endpoint but `store.set()` fails | `api.py:214-220`: no transaction/rollback | New token obtained, old token still in store, caller thinks refresh succeeded — silent inconsistency | Persist new credential first in temp key, then swap atomically |

## Security Findings

| # | Severity | Impact | Exploitability | Recommendation | Spec Reference |
|---|---|---|---|---|---|
| 1 | **MEDIUM** | EncryptedFileStore parent directory inherits default umask permissions | Local attacker can list directory contents, discover credential store existence | `os.chmod(parent, 0o700)` after `mkdir()` | §13.1 |
| 2 | **LOW** | `code_verifier` not in redaction param/key sets (`_REDACTED_PARAMS`, `_REDACT_DICT_KEYS`) | If logged, PKCE verifier appears in plaintext — enables token exchange if auth code also intercepted | Add `"code_verifier"` to both sets | §13.3 |
| 3 | **LOW** | `SO_REUSEADDR` allows port hijacking on Windows | Malicious local process could bind to OAuth callback port | Consider `SO_EXCLUSIVEADDRUSE` on Windows | §13.2 |
| 4 | **LOW** | Metadata fingerprint enables cross-machine credential correlation | 64-bit SHA-256 prefix identifies same credential across machines | Use HMAC with device-specific key, or document risk in SECURITY.md | §4.3 |
| 5 | **LOW** | `authlm env` writes secrets to stdout — undocumented risk vector | Shell history, terminal scrollback, `ps` can capture | Add note to SECURITY.md under "does NOT protect against" | §13.2 |

## Architecture Issues

| # | Severity | Issue | Location | v0.2.0 Impact |
|---|---|---|---|---|
| 1 | **HIGH** | `api.connect()` uses isinstance branches on concrete method types for callback injection | `api.py:177-184` | Plugin methods cannot inject callbacks without modifying api.py |
| 2 | **HIGH** | Provider registry is a hardcoded module-level list — no registration mechanism | `registry.py:18-28` | Plugins cannot register providers |
| 3 | **HIGH** | Auth table is a module-level global dict — no injection path | `_auth_table.py:24-77` | Plugins cannot add OAuth config for new providers |
| 4 | **MEDIUM** | `Provider` Protocol has no constructor contract — each provider has different `__init__` | `providers/base.py:59`, each provider file | No factory pattern for plugin discovery |
| 5 | **MEDIUM** | Client ID override mechanism only supports 3 hardcoded providers | `_auth_table.py:5-11` | Plugin OAuth providers won't get env var client ID override |
| 6 | **LOW** | `api.py` imports private `_auth_table` module directly | `api.py:14` | Encapsulation debt |

## Nice-to-Have Improvements (Do NOT Block Release)

- Extract shared store-resolution pattern from 5 CLI files into `cli/_context.py` helper
- Add `TypedDict` definitions for device-code and token-endpoint response JSON to reduce `dict[str, Any]`
- Add `DEFAULT_REFRESH_MARGIN = timedelta(minutes=5)` constant for discoverability
- Add Pydantic `Field(description=...)` on credential and metadata model fields
- Add read-once cache to `MetadataStore._read()` based on file mtime
- Replace `assert oauth is not None` with explicit `AuthLMError` raise in provider files (survives `python -O`)
- Add `__version_tuple__` alongside `__version__` for programmatic version comparison
- Merge two sequential `model_copy()` calls in `api.connect()` into one
- Add `get_user_data_path()` to top-level `authlm.__all__` for discoverability
- Consider `SO_EXCLUSIVEADDRUSE` on Windows for PKCE loopback hardening
- Normalize `validate()` docstring to Google-style Args/Returns/Raises (currently uses inline prose)
- Add `validate()` to the `ConnectionMethod.connect()` Protocol's signature — it's unused in implementations (dead `store` param)
- Silence `SO_REUSEADDR` concern by binding loopback server with explicit `SO_EXCLUSIVEADDRUSE` on Windows

## Final Verdict

### Is AuthLM v0.1.0 ready to release?

**Yes, after 2 P0 and 2 P1 fixes.** The implementation is fully compliant with the v0.1.0 specification. Code quality is production-grade. Security engineering is strong. The test suite covers 13/15 spec-required behaviors. The package build configuration and CI pipeline are correct.

### What blocks the release?

Two zero-effort documentation fixes and two low-effort code fixes:

| Priority | Action | Effort | Files |
|---|---|---|---|
| **P0** | Fix README code examples — add `method_id="api_key"` to both `connect()` calls | 5 min | `README.md` |
| **P0** | Fix AGENTS.md — update `_context.py` description to `get_metadata_path()` (remove phantom `get_store()` and `is_tty()`) | 2 min | `AGENTS.md` |
| **P1** | Add `timeout=30.0` to two missing HTTP POST calls (`oauth_device.py:_poll_for_token`, `oauth_pkce.py:_exchange_code`) | 5 min | `oauth_device.py`, `oauth_pkce.py` |
| **P1** | Fix `httpx.AsyncClient` leak in `connection_methods()` — inject shared client or close after connect | 30 min | `providers/openai.py`, `providers/anthropic.py`, `providers/google.py` |

### What should be fixed first?

1. P0 documentation fixes (blocking user onboarding)
2. P1 code fixes (defense against network hangs and resource leaks)
3. P2 security hardening in v0.1.1 (parent dir permissions, documented env vars)
4. P3 test gap closures in v0.1.1 (concurrent refresh, store.set failure)
5. Architecture redesign for v0.2.0 plugin system (registry, auth table, callback injection)

### What can wait until later versions?

All architecture issues (hardcoded registry, isinstance branches, Protocol constructor contracts) are acceptably scoped for v0.1.0's "first-party only" design. The v0.2.0 plugin system work should begin with a dedicated architecture refactor of the provider registry, auth table, and callback-injection mechanism before any plugin code is written.
