# AuthLM v0.1 — Design Specification

**Version:** 0.1.0
**Status:** Design (pre-implementation)
**Date:** 2026-06-25
**License:** Apache-2.0

---

## 1. Overview

**AuthLM** is a Python library for managing authentication and credentials for AI providers (OpenAI, Anthropic, Google, GitHub Copilot, xAI, Poe, OpenRouter, Mistral, and 30+ others). It is **auth-only**: it does not generate text, route requests, choose models, or replace LiteLLM. Its job ends when it can confidently say: *"This user has successfully connected `<provider>` via `<method>`."*

**Positioning:** AuthLM is the secure, OS-keychain-backed credential layer for AI providers. The first phase targets CLI tools and local-first AI applications. The second phase expands to a broader "connect provider" UX for consumer applications.

**Differentiators:**
- **Auth-only** — separation of concerns; composes with any inference library.
- **OS keychain by default** — never plaintext (unlike `llm keys` JSON storage).
- **Plugin architecture** — provider ecosystem via Python entry points.
- **Pluggable CredentialStore** — swap keychain for Vault, Bitwarden, encrypted file.

**Non-goals:**
- Not an inference library. Does not call provider APIs (except for validation probes).
- Not a model router. Use LiteLLM or `llm` for that.
- Not a SaaS-integration platform. Use Nango or Composio for SaaS auth.
- Not a secret broker for running agents. Use Infisical Agent Vault for that.
- Not a session-extraction tool by default. The library is neutral on auth policy; integrators bear responsibility for ToS compliance with any method they choose to expose.

## 2. Architecture

### 2.1 Three-layer model

```
┌─────────────────────────────────────────────────────────────┐
│  Consumer (CLI tool, web app, agent framework)              │
└────────────────────────┬────────────────────────────────────┘
                         │ imports
┌────────────────────────▼────────────────────────────────────┐
│  Public API: get_credential(), get_valid_credential(),     │
│              refresh(), should_refresh(), connect()         │
└────────────────────────┬────────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────────┐
│  Provider → ConnectionMethod → Credential → CredentialStore│
│  (registry)    (auth flow)          (typed)      (pluggable)│
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Core abstractions

| Abstraction | Role | Lifetime |
|---|---|---|
| `Provider` | Identifies an AI service (OpenAI, Anthropic, ...) | Singleton, registered |
| `ConnectionMethod` | A specific way to authenticate to a provider | Per-provider, immutable |
| `Credential` | A validated, typed authentication artifact | Per-store-entry |
| `CredentialStore` | Pluggable persistence (keychain, file, env, ...) | Singleton, swappable |
| `AuthLMError` hierarchy | Typed exceptions for all failure modes | N/A |

### 2.3 Design principles

1. **Pure auth, pure library.** No inference, no UI, no telemetry.
2. **Pluggable everything.** Stores, providers, and connection methods are all pluggable via entry points.
3. **Explicit over implicit.** No magic auto-refresh on every read; consumer chooses semantics.
4. **Secure by default.** OS keychain; secrets never in plaintext, logs, or exception messages.
5. **Honest about limits.** AuthLM protects secrets at rest and in transit to the store. It cannot protect against a compromised host, in-process exfiltration, or malicious provider SDKs.
6. **Async at the public API boundary; sync for local I/O and pure computation.** All public API functions (`get_credential`, `get_valid_credential`, `refresh`, `validate`, `connect`) are `async`, even when they only call sync stores — this gives consumers one consistent mental model and keeps their event loop responsive during network I/O. Internals that never touch the network stay sync: `CredentialStore.*` (OS keychain, file, env, memory are all sub-ms local calls), `should_refresh()` (pure datetime arithmetic), and `backend_name()` (pure return). The python-conventions rule against async-for-sequential-code applies to intra-function concurrency, not to I/O-bound public APIs that must not block a consumer's event loop. The CLI is a thin sync wrapper that calls `asyncio.run()` on async library functions.

## 3. Core Data Model

### 3.1 Credential types

`Credential` is a Pydantic discriminated union. All credentials share a `provider: str` and `alias: str` for identification.

```python
from __future__ import annotations

from datetime import datetime
from typing import Literal
from pydantic import BaseModel, Field


class Credential(BaseModel):
    provider: str
    alias: str
    method_id: str
    warning_acknowledged_at: datetime | None = None


class ApiKeyCredential(Credential):
    type: Literal["api_key"] = "api_key"
    secret: str


class OAuthCredential(Credential):
    type: Literal["oauth"] = "oauth"
    access_token: str
    refresh_token: str | None
    expires_at: datetime | None
    scopes: list[str] = Field(default_factory=list)
    client_id: str | None = None  # for refresh-token-rotation providers


class AwsCredential(Credential):
    type: Literal["aws"] = "aws"
    access_key_id: str
    secret_access_key: str
    session_token: str | None = None


class AzureAdCredential(Credential):
    type: Literal["azure_ad"] = "azure_ad"
    tenant_id: str
    client_id: str
    client_secret: str | None
    # device-code or federated flows store access_token + refresh_token inline
    access_token: str | None = None
    refresh_token: str | None = None
    expires_at: datetime | None = None
```

**Design notes:**
- The `type` field enables Pydantic discrimination; consumers pattern-match on it.
- `OAuthCredential.refresh_token` is `None` for providers that don't issue one (rare; most do).
- `expires_at` uses UTC-aware `datetime` objects; consumers must handle `None` (no expiry).
- `warning_acknowledged_at` records when the user confirmed the connection-method warning. Stores track this for audit.

**v0.1.0 scope:** Only `ApiKeyCredential` and `OAuthCredential` are **implemented in v0.1.0**. The 4 auth-required built-in providers (OpenAI, Anthropic, Google, OpenRouter) all produce one of these two types. `OllamaProvider` (no-auth edge case) is **deferred to a later milestone** because the `AuthMethod.NONE` enum value it requires does not appear elsewhere in v0.1.0; introducing it now would be speculative. `AwsCredential` and `AzureAdCredential` are **planned for v0.2.0** when `bedrock` and `azure` providers are added. They are included in this spec to lock the discriminated union shape, but no code, tests, or stores for them ship in v0.1.0.

### 3.2 Provider

```python
from __future__ import annotations

from collections.abc import Sequence
from typing import Protocol


class Provider(Protocol):
    @property
    def id(self) -> str:
        """Stable identifier, e.g. 'openai', 'anthropic'."""

    @property
    def display_name(self) -> str:
        """Human-readable name, e.g. 'OpenAI'."""

    @property
    def docs_url(self) -> str | None:
        """Link to the provider's API key signup page."""

    @property
    def logo_url(self) -> str | None:
        """Optional logo for CLI/UI rendering."""

    def connection_methods(self, *, include_warned: bool) -> Sequence[ConnectionMethod]:
        """Return the connection methods this provider supports.
        Methods with a non-None warning are excluded unless include_warned=True.
        """
```

**Built-in providers** (shipped in the core wheel, activated by SDK availability):

AuthLM itself uses `httpx` for all auth operations (OAuth flows, validation probes). Provider SDKs are **not required by AuthLM** — they are listed as extras for consumer convenience only. The "SDK" column below indicates which package a consumer would install for inference with that provider; AuthLM works without any of them.

v0.1.0 ships 4 auth-required providers that together exercise every connection-method type in the architecture (API key, PKCE OAuth, device-code OAuth, warned OAuth, legitimate OAuth). The no-auth edge case (Ollama) is deferred to a later milestone alongside the `AuthMethod.NONE` enum value it requires. Long-tail providers are deferred to v0.2.0.

| Provider ID | Display Name | SDK (consumer convenience) | Methods in v0.1.0 |
|---|---|---|---|
| `openai` | OpenAI | extras: `openai` | `api_key`, `chatgpt_oauth_browser`, `chatgpt_oauth_device` |
| `anthropic` | Anthropic | extras: `anthropic` | `api_key`, `claude_pro_oauth_browser`, `claude_pro_oauth_device` |
| `google` | Google AI | extras: `google-genai` | `api_key`, `oauth_browser` |
| `openrouter` | OpenRouter | extras: `openai` (OpenAI-compatible) | `api_key` |

**What this set proves before scaling:**
- API key storage + retrieval (`openai`, `anthropic`, `google`, `openrouter`)
- OAuth PKCE browser flow (`openai` ChatGPT, `anthropic` Claude Pro, `google` AI Studio)
- OAuth device-code flow (`openai` ChatGPT)
- Warning-gated method — grey-area OAuth (`anthropic` Claude Pro/Max)
- Legitimate OAuth without warning (`google` AI Studio)
- OpenAI-compatible aggregator (`openrouter`)

**Deferred from v0.1.0:**
- `ollama` (Ollama, no-auth) — needs `AuthMethod.NONE` enum extension. Tracked as a follow-up; the no-auth semantic is straightforward and the implementation cost is small once the enum value exists.

### 3.3 ConnectionMethod

```python
from __future__ import annotations

from collections.abc import Sequence
from typing import Protocol


class ConnectionMethod(Protocol):
    @property
    def id(self) -> str:
        """Unique within the provider, e.g. 'api_key', 'chatgpt_oauth_browser'."""

    @property
    def label(self) -> str:
        """What the user sees in the CLI, e.g. 'Manually enter API key'."""

    @property
    def warning(self) -> str | None:
        """If non-None, printed before connect; user must confirm.
        Used for methods where the provider has explicitly prohibited third-party use.
        """

    @property
    def oauth_grant(self) -> OAuthGrant | None:
        """None for non-OAuth methods; OAuthGrant enum for OAuth flows."""

    async def connect(self, *, store: CredentialStore) -> Credential:
        """Run the auth flow interactively; return a credential with
        `provider`, `alias="default"`, and `method_id=self.id` populated.

        **Persistence is the caller's responsibility.** The method does
        not call `store.set()` itself. The `authlm.api.connect()`
        orchestrator persists the returned credential and updates
        metadata; tests and other low-level callers can persist
        themselves. Splitting persistence out of the method keeps
        `connect()` side-effect-free except for network I/O and keeps
        the method testable in isolation.
        """

    async def validate(self, cred: Credential, *, force: bool) -> bool:
        """Probe whether the credential is currently usable.
        Refuses if self.warning is set and force=False.
        """
```

**`OAuthGrant`** enumerates the two standard OAuth flows:

```python
from enum import StrEnum


class OAuthGrant(StrEnum):
    AUTHORIZATION_CODE_PKCE = "authorization_code_pkce"  # browser-based
    DEVICE_CODE = "device_code"                           # headless
```

**Connection-method reference** (v0.1.0 built-in providers; community plugins and v0.2.0 providers add more):

| Provider | Method ID | Label | Grant | Warning |
|---|---|---|---|---|
| `openai` | `api_key` | Manually enter API key | — | — |
| `openai` | `chatgpt_oauth_browser` | ChatGPT Pro/Plus (browser) | PKCE | — |
| `openai` | `chatgpt_oauth_device` | ChatGPT Pro/Plus (headless) | DEVICE_CODE | — |
| `anthropic` | `api_key` | Manually enter API key | — | — |
| `anthropic` | `claude_pro_oauth_browser` | Claude Pro/Max (browser) | PKCE | **Anthropic prohibits this in their ToS...** |
| `anthropic` | `claude_pro_oauth_device` | Claude Pro/Max (headless) | DEVICE_CODE | **Anthropic prohibits this in their ToS...** |
| `google` | `api_key` | Manually enter API key | — | — |
| `google` | `oauth_browser` | Google AI Studio (browser) | PKCE | — |
| `openrouter` | `api_key` | Manually enter API key | — | — |
| ~~`ollama`~~ | ~~(none)~~ | ~~(local, no auth)~~ | — | — | _Deferred from v0.1.0 — needs `AuthMethod.NONE` enum extension (see §3.2)._ |

**Warning policy:**
- A method has `warning: str | None`. If non-None, the CLI prints it and asks `[y/N]` before connecting.
- `validate()` refuses to probe warned methods unless `force=True` is passed.
- The library never refuses to `connect()` a warned method if the user confirms. The integrator decides policy.
- Currently the **only** methods with warnings are Anthropic's Claude Pro/Max (explicitly prohibited by Anthropic, removed from opencode v1.3.0).

## 4. CredentialStore Abstraction

### 4.1 Protocol

```python
from __future__ import annotations

from collections.abc import Iterator
from typing import Protocol


class CredentialStore(Protocol):
    """Pluggable persistence for credentials.
    Default implementation uses the OS keychain via `keyring`.
    """

    def get(self, provider: str, alias: str) -> Credential | None:
        """Retrieve a credential, or None if not found."""

    def set(self, credential: Credential) -> None:
        """Persist a credential, overwriting any existing entry for the same (provider, alias)."""

    def delete(self, provider: str, alias: str) -> bool:
        """Delete a credential; return True if it existed."""

    def list(self) -> Iterator[tuple[str, str]]:
        """Yield (provider, alias) pairs for all stored credentials."""

    def backend_name(self) -> str:
        """Human-readable name of the active backend, e.g. 'macOS Keychain'."""
```

### 4.2 Built-in backends

| Backend | Class | Use case |
|---|---|---|
| OS keychain | `KeyringStore` | Default. macOS Keychain / Windows Credential Manager / Linux Secret Service via the `keyring` library. |
| Encrypted file | `EncryptedFileStore` | Fallback for headless/CI when no keyring is available. Uses `cryptography.fernet`; key derived from a user passphrase or OS-protected secret. |
| Environment | `EnvStore` | Read-only; reads from env vars (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, etc.). Useful for CI, Docker, ephemeral environments. Never writes. |
| Memory | `MemoryStore` | In-process, for tests. Cleared on process exit. |

**Backend selection:**
- AuthLM auto-selects: `KeyringStore` if a keyring backend is available, else `EncryptedFileStore`, else `EnvStore` (with a printed warning that credentials won't persist).
- Users can override: `AUTHLM_STORE=encrypted_file authlm connect openai` or programmatically via `authlm.set_store(...)`.
- The active backend is reported by `authlm status --backend`.

### 4.3 Metadata storage

The store holds **secrets**. Non-secret metadata (provider display name, last-validated timestamp, warning-acknowledged timestamp, OAuth client_id for diagnostics) lives in a separate JSON file managed by `authlm`:

- Location: `~/.local/share/authlm/metadata.json` (XDG-compliant via `platformdirs`).
- Schema: `dict[str, dict[str, MetadataEntry]]` keyed by `(provider, alias)`.
- Fingerprints (SHA-256 of the secret, truncated) for change detection — never the secret itself.
- `MetadataEntry` includes: `provider_display_name`, `method_id`, `connected_at`, `last_validated_at`, `warning_acknowledged_at`, `scopes` (for OAuth).

This split mirrors `row-bot`'s design: keychain for secrets, JSON for metadata.

### 4.4 Plugin stores

Third-party stores (HashiCorp Vault, Bitwarden, 1Password, AWS Secrets Manager) register via entry points:

```toml
[project.entry-points."authlm.stores"]
vault = "authlm_vault:VaultStore"
```

The plugin's store class implements the `CredentialStore` Protocol. Users select via `AUTHLM_STORE=vault` or programmatically.

## 5. Token Refresh

### 5.1 Public API

```python
from __future__ import annotations

from datetime import datetime, timedelta, timezone


async def get_credential(
    provider: str,
    *,
    alias: str,
) -> Credential:
    """Return the stored credential as-is, even if expired.
    Fast, no network. Use when you need the raw value and will check expiry yourself.
    Raises CredentialNotFound if no credential is stored for (provider, alias).
    """


async def get_valid_credential(
    provider: str,
    *,
    alias: str,
    refresh_margin: timedelta,
) -> Credential:
    """Return a credential that is currently usable.
    If the stored credential is expired or within refresh_margin of expiry,
    attempts a refresh via the provider's token endpoint.
    Makes a network call only when refresh is needed.
    Raises CredentialNotFound if no credential is stored for (provider, alias).
    """


async def refresh(
    provider: str,
    *,
    alias: str,
) -> Credential:
    """Force-refresh the credential regardless of expiry.
    Returns the new credential. Raises RefreshFailed on transient errors,
    ReconnectionRequired if the refresh token is dead.
    """


def should_refresh(cred: Credential, *, margin: timedelta) -> bool:
    """Return True if the credential is expired or within margin of expiry.
    For credentials without expires_at (e.g. ApiKeyCredential), returns False.
    """
```

**Note on default parameter values:** Per the project's coding conventions (§14), no function parameters have default values. Callers must pass all arguments explicitly. The recommended refresh margin is 5 minutes; callers pass `timedelta(minutes=5)` explicitly. Pydantic model fields are exempt from this rule (optional fields require defaults).

**Error hierarchy:**

```python
class AuthLMError(Exception):
    """Base for all AuthLM exceptions."""

class CredentialNotFound(AuthLMError):
    """No credential stored for the given (provider, alias)."""

class RefreshFailed(AuthLMError):
    """Transient network error or 5xx from token endpoint. Retry with backoff."""

class ReconnectionRequired(AuthLMError):
    """Refresh token is dead (invalid_grant). User must re-run connect()."""

class AccessDenied(AuthLMError):
    """Token works but user lacks access (403 entitlement_denied)."""

class TokenEndpointError(AuthLMError):
    """Other 4xx from token endpoint. Surface provider's error message."""

class ProviderNotAvailable(AuthLMError):
    """Provider SDK not installed. Message includes install hint."""

class AliasCollisionError(AuthLMError):
    """Two plugins registered the same (provider, alias)."""
```

### 5.2 Refresh token rotation

Some providers (OpenAI) rotate the refresh token on each use. The refresh implementation must:

1. POST to the provider's token endpoint with the stored `refresh_token`.
2. Parse the response for `access_token` and (if present) a new `refresh_token`.
3. Persist **both** to the store, overwriting the old pair.
4. If the response omits a new refresh token, keep the existing one.

Failure to rotate correctly causes a hard failure on the *next* refresh. This is a common bug in hand-rolled OAuth clients; AuthLM's centralization prevents it.

### 5.3 Error taxonomy

| Exception | Cause | Consumer action |
|---|---|---|
| `RefreshFailed` | Transient network error or 5xx from token endpoint | Retry with backoff |
| `ReconnectionRequired` | `invalid_grant` / `expired_refresh_token`; refresh token is dead | User must re-run `connect()` |
| `AccessDenied` | `entitlement_denied` / 403; token works but user lacks access | Surface "your account doesn't have access to this model" |
| `TokenEndpointError` | Other 4xx from token endpoint | Surface provider's error message |

These are distinct user-facing states, not generic "refresh failed."

### 5.4 Concurrency

v1 assumption: **single-process.** A credential library used by multi-process apps could see two processes refresh simultaneously, invalidating each other's refresh tokens. This is documented as a known limitation; v2 may add file-locking or a refresh daemon.

## 6. Multi-Credential Support

Every credential is keyed by `(provider, alias)`. The default alias is `"default"`.

```bash
# First OpenAI key
authlm connect openai --alias personal

# Second OpenAI key (work account)
authlm connect openai --alias work

# Use a specific alias
authlm env openai --alias work
```

```python
cred = authlm.get_credential("openai", alias="work")
```

The default alias is `"default"`; callers who want the conventional default pass `alias="default"` explicitly. `authlm list` shows all aliases per provider. `authlm status openai` shows all aliases for that provider.

## 7. Validation

```python
async def validate(
    cred: Credential,
    *,
    force: bool,
) -> bool:
    """Probe whether a credential is currently usable.
    Makes a lightweight API call (e.g. GET /v1/models).
    Returns True if the credential works, False if it doesn't.

    Delegates to the ConnectionMethod that produced this credential:
    looks up the method by cred.method_id on the cred's provider, then
    calls method.validate(cred, force=force).
    """
```

**Policy:**
- `validate()` is **never automatic.** It is only called when the consumer (CLI or library user) explicitly requests it.
- For methods with a `warning`, `validate()` raises `PermissionError` unless `force=True` is passed. `PermissionError` (the built-in) is the right signal here: the library is refusing to act without explicit user consent, distinct from operational errors like `TokenEndpointError` or `RefreshFailed`. Callers that need to catch this alongside other authlm errors should use `except (AuthLMError, PermissionError)`. Even with `force=True`, the M3 CLI prints an additional warning that the validation call itself may be detectable by the provider.
- The CLI command is `authlm status <provider> --validate` which calls `validate()` for the default alias. `authlm status <provider> --all --validate` validates all aliases. Neither is called automatically on `connect()` or `list()`.

## 8. Provider Registry and models.dev Integration

### 8.1 models.dev as metadata source

AuthLM uses [models.dev](https://github.com/anomalyco/models.dev) (MIT, 5.2k stars) as a read-only provider metadata source:

- **What models.dev provides:** provider IDs, display names, docs URLs, env-var names, SDK package IDs, base URLs for OpenAI-compatible providers, logos, model lists with pricing/capabilities.
- **What models.dev does NOT provide:** OAuth endpoints, scopes, client IDs, token URLs, key-issuance URLs, credential-type taxonomy, header-placement rules, refresh logic. AuthLM owns all of this.
- **Integration pattern:**
  1. At first run (or on explicit `authlm refresh-registry`), fetch `https://models.dev/api.json`.
  2. Cache the JSON locally at `~/.local/share/authlm/models-dev-cache.json`.
  3. Refresh daily; never fetch per-auth-call.
  4. Vendor a snapshot of the JSON in the wheel (`src/authlm/_vendor/models-dev-snapshot.json`) as a fallback for offline installs.
- **Join key:** provider ID from models.dev. AuthLM's own auth table is keyed by the same ID.
- **AuthLM's own auth table** (`src/authlm/_auth_table.py`): for each provider ID, the auth specifics models.dev can't give us — `credential_type`, `oauth_config` (endpoints, scopes, client_id), `key_url`, `header_rules`, `refresh_endpoint`, `validation_endpoint`.

**v0.1.0 status: scaffolding only.** The `models_dev.py` module (fetch, cache, vendored fallback) is implemented and tested, but **not wired into providers**. The 4 built-in providers hardcode their `display_name`, `docs_url`, and `logo_url` instead of calling `get_provider_metadata()`. The join against models.dev's `api.json` is deferred to v0.2.0, when the provider count grows from 4 to 20+ and the network/cache/fallback chain justifies itself. Hardcoding 4 providers' metadata is simpler, faster, and avoids a first-run network call — the right tradeoff for v0.1.0's scale. The module is ready for v0.2: wiring is mechanical (providers call `get_provider_metadata(self.id, cache_path=...)` instead of hardcoding).

### 8.2 Schema for the auth table

```python
from __future__ import annotations

from pydantic import BaseModel, HttpUrl


class OAuthConfig(BaseModel):
    authorize_url: HttpUrl
    token_url: HttpUrl
    device_code_url: HttpUrl | None = None
    client_id: str
    default_scopes: list[str]
    # Loopback port range for PKCE redirect; per-provider.
    loopback_port: int | None = None
    # Whether the provider rotates refresh tokens.
    rotates_refresh_token: bool = False


class AuthTableEntry(BaseModel):
    provider_id: str
    credential_types: list[str]  # which Credential subclasses are valid
    # OAuth flow config (None if no OAuth method)
    oauth: OAuthConfig | None = None
    # Lightweight endpoint for validation probes
    validation_url: HttpUrl | None = None
    # Override models.dev's docs_url with a more specific key-issuance page
    key_issuance_url: HttpUrl | None = None
```

The auth table is a Python module (not a JSON file) because the OAuth config is too sensitive to distribute via a public data catalog and because it needs to be importable for type checking.

**Public OAuth client IDs** are hardcoded in `_auth_table.py` for the three OAuth-capable built-in providers (OpenAI, Anthropic, Google) using publicly-known client IDs from official CLI tools:

| Provider | Default `client_id` | Source |
|---|---|---|
| `openai` | `app_EMoamEEZ73f0CkXaXp7hrann` | OpenAI Codex CLI public app |
| `anthropic` | `9d1c250a-e61b-44d9-88ed-5944d1962f5e` | Anthropic Claude Code CLI public app |
| `google` | `681255809395-oo8ft2oprdrnp9e3aqf6av3hmdib135j.apps.googleusercontent.com` | Google AI Studio installed-app client |

These match the approach used by `llm`, opencode, and the official CLI tools — distributing a public client ID is the same as distributing the official tool's app, and the values are public.

**Env-var override** is supported for users who register their own OAuth app (e.g., for an enterprise environment that needs a custom redirect URI):

- `AUTHLM_OPENAI_CLIENT_ID`
- `AUTHLM_ANTHROPIC_CLIENT_ID`
- `AUTHLM_GOOGLE_CLIENT_ID`

If the env var is set, it takes precedence over the hardcoded default. If unset, the default is used. Documented in `AGENTS.md` and the README.

## 9. Plugin System

### 9.1 Why pluggy

AuthLM uses [pluggy](https://github.com/pytest-dev/pluggy) (the same plugin engine as `pytest`, `tox`, and `llm`) for its plugin system. pluggy provides:

- Hook specification and implementation markers with signature validation.
- Plugin isolation (a broken plugin doesn't kill the core).
- Lazy loading via entry points.
- Battle-tested in production by thousands of projects.

### 9.1a First-party providers vs third-party plugins — one PyPI package

**First-party providers (maintained by the AuthLM team) ship in the `authlm` wheel.** No separate PyPI packages. When a new provider is added in a new release, the provider code goes into `src/authlm/providers/` and a corresponding extra is added to `pyproject.toml` if it needs an SDK dep. The user experience is:

```
pip install authlm                    # base install + 5 v0.1 providers
pip install --upgrade authlm          # new httpx-only providers work immediately
pip install "authlm[mistral]"         # installs mistralai SDK, activates Mistral provider
pip install "authlm[all]"             # installs all provider SDKs
```

httpx-only providers (GitHub, xAI, Poe, Cerebras, Perplexity, HuggingFace, Moonshot, MiniMax, GitLab, DigitalOcean) need **no extra** — they activate immediately after `pip install --upgrade authlm` because their only network dependency is `httpx`, which is already a core dep.

SDK-backed providers (OpenAI, Anthropic, Google, Mistral, Groq, etc.) require the corresponding extra to install the provider's SDK. The provider code is always present in the wheel; the extra just installs the SDK dep. If the user tries to connect without the extra, the provider raises `ProviderNotAvailable` with a helpful install hint.

**PyPI pages the maintainer owns: one.** The `authlm` package. The extras list grows; the package count stays at 1.

**Third-party community plugins** (maintained by the community, not by the AuthLM team) are separate PyPI packages — e.g. `authlm-deepinfra`. These use the same `authlm` entry-point group. The AuthLM maintainer does **not** own, publish, or support these packages. They are optional and community-driven. §15.1 (CONTRIBUTING.md "How to add a provider" section) documents how to create one.

The distinction:

| | First-party (you maintain) | Third-party (community maintains) |
|---|---|---|
| Where the code lives | `src/authlm/providers/` in the `authlm` wheel | Separate repo, separate PyPI package |
| How users install | `pip install "authlm[<extra>]"` or just `pip install --upgrade authlm` (httpx-only) | `pip install authlm-<name>` |
| PyPI pages the maintainer owns | **1** (`authlm`) | **0** |
| Entry-point registration | Loaded as a default plugin via `DEFAULT_PLUGINS` | Loaded via `authlm` entry-point group |
| Support responsibility | AuthLM maintainers | The community package author |

### 9.2 Hookspecs

```python
# src/authlm/hookspecs.py
from __future__ import annotations

from pluggy import HookspecMarker

hookspec = HookspecMarker("authlm")


@hookspec
def register_providers(register) -> None:
    """Register Provider instances.
    ``register(provider: Provider)`` adds a provider to the registry.
    """


@hookspec

    """Register ConnectionMethod instances.
    ``register(method: ConnectionMethod, provider_id: str)`` adds a method.
    """
```

### 9.3 Plugin loading

```python
# src/authlm/plugins.py
from __future__ import annotations

import importlib
import sys

import pluggy

from . import hookspecs

pm: pluggy.PluginManager | None = None

_loaded = False
DEFAULT_PLUGINS: tuple[str, ...] = (
    "authlm.providers.openai",
    "authlm.providers.anthropic",
    # ... all built-in providers
)


def load_plugins() -> None:
    global _loaded, pm
    if _loaded:
        return
    _loaded = True
    pm = pluggy.PluginManager("authlm")
    pm.add_hookspecs(hookspecs)
    if not getattr(sys, "_called_from_test", False):
        pm.load_setuptools_entrypoints("authlm")
    for plugin_name in DEFAULT_PLUGINS:
        module = importlib.import_module(plugin_name)
        pm.register(module, plugin_name)
```

### 9.5 Plugin safety rules

1. **Lazy imports.** Plugins must import their SDKs inside `register_*` or `connect()`, not at module top level. This prevents forcing SDK installs on users who don't use that plugin.
2. **Test isolation.** In tests, `sys._called_from_test = True` skips entry-point autoload, so installed third-party plugins don't pollute the test suite.
3. **Isolation.** A broken plugin's exception is caught, logged with the plugin name, and does not prevent other plugins from loading.
4. **Alias collision.** If two plugins register the same `(provider, alias)`, AuthLM raises a loud error during registration. Silent shadowing is a security bug in a credential library.

## 10. CLI Design

### 10.1 The five commands

The CLI is a thin consumer of the library. It uses Click.

```
authlm connect <provider> [--alias <name>] [--method <method_id>]
authlm list
authlm status [<provider>] [--alias <name>] [--all]
authlm disconnect <provider> [--alias <name>]
authlm env <provider> [--alias <name>] [--export-format shell|docker|github]
```

### 10.2 Command semantics

**`authlm connect <provider>`**
1. Look up the provider.
2. List its connection methods (filtered by `warning` unless `--include-warned`).
3. User picks a method.
4. If the method has a `warning`, print it and ask `[y/N]`.
5. Run the method's `connect()` flow interactively.
6. Store the resulting credential.
7. Update metadata.

**`authlm list`**
Print a table: `Provider | Alias | Method | Backend | Last Validated`.

**`authlm status [<provider>]`**
For the default alias (or all aliases with `--all`): print credential metadata, backend, and (if `--validate`) call `validate()`. Validation is explicit; without `--validate`, no network call.

**`authlm disconnect <provider> [--alias <name>]`**
Delete the credential and its metadata entry. Print a confirmation (unless `--yes`).

**`authlm env <provider> [--alias <name>]`**
Export the credential's secret to stdout in the requested format. For `api_key` credentials, outputs `PROVIDER_API_KEY=...`. For `oauth`, outputs `PROVIDER_ACCESS_TOKEN=...` (and `PROVIDER_REFRESH_TOKEN=...` if available). Designed for use in shell: `eval "$(authlm env openai)"`.

### 10.3 Two additional commands (out of v0.1, planned v0.2)

**`authlm export [--format json|env] [--output <file>]`**
Exports all stored credentials to a portable file. Secrets are **not** encrypted in the export (the export file is plaintext). Users are warned. This is for migration and disaster recovery, not for sharing.

**`authlm import llm`**
Reads `llm`'s key store (~/.config/io.datasette.llm/keys.json or similar), copies keys into AuthLM's store, and reports what was imported. One-way migration.

Both are deferred to v0.2 to keep v0.1 scope tight.

## 12. Testing Strategy

### 12.1 Test layers

| Layer | Tool | Purpose |
|---|---|---|
| Unit | `respx` | Fast, explicit route mocks for OAuth/token-refresh logic |
| Contract | `pytest-recording` (VCR.py) | Recorded provider responses; replayed offline |
| Snapshot | `syrupy` | Assert stable CLI output and JSON serialization |
| Security | `bandit` (via `ruff`), manual review | Catch secret leaks, unsafe deserialization |

### 12.2 Key fixtures (`tests/conftest.py`)

- `sys._called_from_test = True` to disable third-party plugin autoload.
- `AUTHLM_USER_PATH` redirected to `tmpdir` so no test touches the real user keychain.
- `PYTHON_KEYRING_BACKEND=keyring.backends.null.Keyring` to prevent OS keychain access in tests.
- `MemoryStore` fixture for unit tests.
- `httpx_mock` fixtures per provider.
- `MockOAuthServer` for testing PKCE flows without hitting real providers.

### 12.3 Cassette hygiene

- **Never commit live tokens.** CI runs a grep step that fails on patterns: `sk-`, `xoxb-`, `ghp_`, `ya29.`, `Bearer [A-Za-z0-9_-]{20,}`.
- All cassettes use `filter_headers=["Authorization"]` and `filter_post_data_parameters=["refresh_token", "code", "client_secret"]`.
- `before_record_response` scrubs `access_token` and `refresh_token` from response bodies.

### 12.4 CI matrix

- **OS:** Ubuntu latest, macOS latest, Windows latest (keychain behavior differs per OS).
- **Python:** 3.11, 3.12, 3.13, 3.14.
- **Lint and type-check on one OS** to save CI minutes.
- **pip-audit** runs on every PR to catch supply-chain vulnerabilities in dependencies.

### 12.5 What to test

- `get_credential()` returns the stored value, expired or not.
- `get_valid_credential()` calls `refresh()` only when expired or within margin.
- `refresh()` persists both new access_token and new refresh_token (rotation).
- `refresh()` raises `ReconnectionRequired` on `invalid_grant`.
- Store round-trips: set, get, delete, list.
- `validate()` refuses warned methods without `force=True`.
- `validate()` with `force=True` still prints a warning.
- Plugin isolation: a broken plugin doesn't crash core.
- Alias collision detection during plugin registration.
- OAuth PKCE: verifier/challenge pairing is correct.
- OAuth device-code: poll interval and timeout behavior.

## 13. Security

### 13.1 SECURITY.md (required sections)

1. **Supported versions** — which minor lines receive security fixes (SemVer-aligned).
2. **Reporting a vulnerability** — GitHub Private vulnerability reporting (Security tab → "Report a vulnerability"). Target SLA: 72h acknowledge, 14d patch.
3. **Threat model** — what AuthLM does and does NOT protect against.
4. **Credential storage backends** — per-backend security notes.
5. **Logging and redaction** — what gets redacted in logs, exceptions, and cassettes.
6. **Coordinated disclosure** — embargo policy, credit policy.
7. **Security-relevant dependencies** — pinned and tracked via Dependabot.

### 13.2 Threat model (explicit)

**AuthLM protects against:**
- Secrets written to disk in plaintext (default backend is OS keychain).
- Secrets appearing in log output or exception messages (redacted).
- Secrets appearing in VCR cassettes (scrubbed).
- Token expiry going unnoticed (explicit `get_valid_credential()` API).
- Refresh-token rotation bugs (centralized implementation).
- Plugin supply-chain attacks via namespace confusion (strict entry-point group).

**AuthLM does NOT protect against:**
- A compromised host or root access (the OS keychain is only as secure as the OS).
- In-process exfiltration by a malicious library you installed (AuthLM hands the secret to your code; what happens after is on you).
- Malicious provider SDKs (a backdoored `openai` package can exfiltrate the key after AuthLM retrieves it).
- Network interception of API calls (use TLS; AuthLM uses `httpx` with HTTPS by default).
- Social engineering or phishing (the user is the attack surface).

### 13.3 Redaction policy

The following are redacted in all log output, exception messages, and cassettes:
- `Authorization` headers (any scheme)
- `access_token`, `refresh_token`, `id_token` fields in JSON
- `api_key`, `secret`, `client_secret` fields in JSON
- Cookie values
- Query parameters named `code`, `token`
- HTTP response body substrings matching `Bearer <token>` and JSON-string fields named `access_token`/`refresh_token`/`id_token`/`client_secret` (enforced by `redact_body` in `authlm.connection_methods._oauth_helpers`)

`state` (the OAuth CSRF token) is **intentionally not redacted**: it is not a credential, redacting it would make OAuth flow debugging harder, and libraries like `authlib` similarly leave it unredacted. Logging `state=...` from authorize URLs and from loopback callbacks is safe and useful for diagnosing flow errors.

This is enforced by a redaction layer in `authlm.connection_methods._oauth_helpers` (`redact_url` for query parameters, `redact_body` for HTTP response bodies) and by the VCR cassette configuration.

### 13.4 GitHub repository security

- **Dependabot** enabled for `pip` and `github-actions` ecosystems.
- **Secret scanning** enabled.
- **CodeQL** enabled.
- **Branch protection** on `main`: require PR review, require CI pass, no direct push.

## 14. Coding Conventions

All Python code follows the rules in the `python-conventions` skill:

- `from __future__ import annotations` at the top of every file.
- Explicit type annotations on all function signatures and module-level variables.
- `collections.abc` over `typing` for generic types.
- `Literal` types for fixed string sets.
- `Self` return type on class methods returning `self`.
- `from typing_extensions import override` and use `@override` on every method that overrides a base class or Protocol method. This is a plugin-heavy library; `@override` catches signature mismatches in third-party subclasses at type-check time.
- Assert before `typing.cast()`.
- Keyword-only arguments for functions with 5+ parameters.
- No default parameter values; make caller intent explicit.
- `__len__`, `__bool__`, `__contains__` must be O(1).
- Lazy computation; no import-time side effects.
- Strict mypy (`mypy --strict`).
- Ruff for linting and formatting.
- Pydantic for all data crossing process, network, config, or serialization boundaries.

**Additional rules specific to AuthLM:**
- No `print()` calls in library code. Use the `logging` module at `DEBUG` level.
- No broad `except Exception: pass`. Catch specific exceptions; let unexpected failures surface.
- No bare `except:`. Ever.
- No `async` without real concurrency. Sequential awaits are a bug.
- Secrets are never passed through `str()` or `repr()` in error messages. Use the redaction layer.
- Plugin modules import their SDKs lazily (inside `register_*` or `connect()`), not at top level.

## 17. Open Questions and Known Limitations

### 17.1 Open questions deferred to v0.2

- **Adapters as maintained API?** v0.1 ships `as_openai_client()` and `as_anthropic_client()` as docs examples (not maintained API surface) to keep the library pure-auth. If user demand materializes, v0.2 promotes them to maintained API with full test coverage and SemVer guarantees. Gating criterion: at least 3 independent user requests for the same adapter.
- **TUI mode for `connect` flow?** v0.1 is CLI-only (Click). A Textual-based TUI is a v0.3 candidate — depends on whether the interactive method-selection flow proves painful enough to warrant a richer UI.

### 17.2 Known limitations (documented in README)

- **Single-process assumption.** Multi-process apps refreshing the same credential concurrently may invalidate each other's refresh tokens. v0.2 adds file-locking.
- **No proxy/middleware support.** AuthLM does not provide a local proxy that injects credentials. If you need that, use a local proxy like LiteLLM with AuthLM as the credential source.
- **No web UI.** v0.1 is CLI + library. A web UI for non-technical users is deferred to v1.0 (alongside the hosted vault).
- **Linux keychain variance.** Linux's keyring support depends on Secret Service being available (gnome-keyring, KWallet). Headless Linux without a Secret Service backend falls back to `EncryptedFileStore` with a user passphrase. This is documented in the README and in the `keyring` error message.
- **Claude Pro/Max OAuth is shipped but warned.** Anthropic explicitly prohibits this in their ToS. Integrators are responsible for compliance. AuthLM provides the capability and the warning; the integrator provides the policy.

## 18. Appendix: Migration from `llm keys`

`llm` stores API keys as plaintext JSON in `~/.config/io.datasette.llm/keys.json`. AuthLM stores them in the OS keychain. A migration command (`authlm import llm`, v0.2) will:

1. Locate `llm`'s key file (checking `LLM_USER_PATH` env var, then XDG defaults).
2. For each `(provider, key)` pair, call `authlm connect <provider> --alias default` with the key pre-filled (skipping the interactive prompt).
3. Report what was imported and what was skipped.
4. **Not delete** the `llm` key file (let the user do that manually after verifying).

This is a one-way import. The reverse (`authlm export llm`) is not planned — once you've migrated, you should use AuthLM as the source of truth.

## 19. Appendix: Glossary

| Term | Definition |
|---|---|
| **Provider** | An AI service that requires authentication (OpenAI, Anthropic, etc.). |

| **Credential** | A validated, typed authentication artifact stored in a CredentialStore. |
| **CredentialStore** | A pluggable persistence backend (OS keychain, encrypted file, env, memory, or community plugin). |
| **Alias** | A label for multiple credentials of the same provider (e.g., `personal` and `work` for two OpenAI accounts). |
| **Plugin** | A third-party package that registers providers, connection methods, or stores via entry points. |
| **Hookspec** | A pluggy-defined interface that plugins implement. |
| **Refresh margin** | The time window before expiry within which `get_valid_credential()` proactively refreshes. Default: 5 minutes. |
| **Metadata** | Non-secret information about a credential (provider name, connected-at, scopes, warning-acknowledged-at) stored in a separate JSON file. |
| **Cassette** | A recorded HTTP interaction (VCR.py format) used in tests to replay provider responses without making real network calls. |

**End of specification.**
