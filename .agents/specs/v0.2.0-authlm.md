# AuthLM v0.2.0 — Extensibility & Ecosystem

**Version:** 0.2.0
**Status:** Planned (outline)
**SemVer:** Pre-release — API may break.
**Prerequisite:** v0.1.0 shipped.

---

## Theme

AuthLM becomes a platform, not just a library. Third-party plugin authors can add providers, connection methods, and stores via setuptools entry points. The provider count grows from 4 to 20+. The doc site launches. Homebrew install is available.

---

## Prerequisite: v0.1.x Architecture Refactors

Before v0.2.0 plugin work begins, the v0.1.x codebase requires six architectural changes.

### A1: Redesign api.connect() callback injection

**Problem:** `api.connect()` uses `isinstance` branches on concrete method types (`APIKeyMethod`, `OAuthPKCEMethod`, `OAuthDeviceCodeMethod`) to inject CLI callbacks. Plugin-defined method types cannot inject callbacks without modifying `api.py`.

**Fix:** Add `inject_callbacks(self, **callbacks: object) -> ConnectionMethod` to the `ConnectionMethod` Protocol. Each method picks out the callback keys it cares about (`secret_prompt`, `on_prompt`, `open_browser`). `api.connect()` calls `method.inject_callbacks(...)` without isinstance.

### A2: Plugin registration mechanism for provider registry

**Problem:** Providers are registered via a hardcoded list in `registry.py:_build_providers()`. No mechanism for third-party providers to register.

**Fix:** Add `register_provider(factory: Callable[..., Provider])` to `registry.py`. Convert the 4 built-in providers to use this registration function. The pluggy `register_providers` hook in v0.2.0 will call `register_provider` for each plugin provider.

### A3: Plugin registration mechanism for auth table

**Problem:** Auth table is a module-level `dict` literal (`AUTH_TABLE`). No injection path for plugin OAuth configs.

**Fix:** Add `register_auth_entry(provider_id: str, entry: AuthTableEntry)` to `_auth_table.py`. Convert the 4 built-in entries to use this registration function. Generalize the client ID env-var override from 3 hardcoded providers to a convention-based pattern (`AUTHLM_{PROVIDER}_CLIENT_ID`).

### A4: Provider constructor contract

**Problem:** Each provider has a different `__init__` signature. No factory pattern for plugin discovery.

**Fix:** Define a `ProviderFactory` callable protocol: `(secret_prompt, http_client) -> Provider`. Change `register_provider` to accept `ProviderFactory` instead of a raw class.

### A5: Wrap store I/O in asyncio.to_thread()

**Problem:** Sync store I/O (`store.get()`, `store.set()`) is called directly from async functions, blocking the event loop.

**Fix:** Wrap store calls in `asyncio.to_thread()` in `api.py`. This keeps the `CredentialStore` protocol sync (correct for non-async contexts) while preventing event-loop stalls for async consumers.

### A6: Remove api.py / validation.py direct imports of _auth_table

**Problem:** Public API modules import from the `_`-prefixed private module `_auth_table`.

**Fix:** Re-export `get_oauth_config` and `get_auth_entry` from `providers/registry.py`. Update imports in `api.py` and `validation.py`.

## Scope

### Plugin system

- **pluggy-based plugin system** — `hookspecs.py` and `plugins.py` restored and wired through `providers/registry.py`. `pm.hook.register_providers(...)` is the actual provider discovery path, not a hardcoded list.
- **Three hookspecs** — `register_providers`, `register_connection_methods`, `register_stores`. Each takes a `register(callable)` callback.
- **First-party providers as default plugins** — the 4 v0.1.0 providers register via `@hookimpl` in their respective modules. Loaded via `DEFAULT_PLUGINS` tuple.
- **Third-party plugins via entry points** — `[project.entry-points."authlm"]` group. `pm.load_setuptools_entrypoints("authlm")` at startup.
- **Plugin safety rules** — lazy SDK imports inside `register_*` or `connect()`, test isolation via `sys._called_from_test`, broken-plugin isolation (caught + logged, not fatal), alias-collision detection (`AliasCollisionError` raised on duplicate `(provider, alias)`).
- **Plugin authoring guide** — published at `docs/plugin-authoring.md`. Includes a copy-paste skeleton for a new provider.
- **`ProviderNotAvailable` error** — restored. Raised when a provider's SDK extra is not installed. Message includes the install hint (e.g., `pip install "authlm[groq]"`).

### Provider expansion

- **Long-tail providers** — added as first-party or community plugins:
  - httpx-only (no SDK needed): GitHub, xAI, Poe, Cerebras, Perplexity, Hugging Face, Moonshot, MiniMax, GitLab, DigitalOcean.
  - SDK-backed: Mistral, Groq, DeepSeek, Cohere, Together, Fireworks.
  - Cloud providers: AWS Bedrock, Azure OpenAI.
- **`OllamaProvider`** — no-auth provider for local inference. Adds `AuthMethod.NONE` enum value to `OAuthGrant` (or a new `AuthMethod` enum). The `ConnectionMethod` for Ollama has `warning=None` and `oauth_grant=None`; `connect()` returns a `Credential` with `method_id="none"` and no secret.
- **`AwsCredential`** — new credential type for AWS Bedrock. Fields: `access_key_id`, `secret_access_key`, `session_token`. Added to `CredentialUnion`.
- **`AzureAdCredential`** — new credential type for Azure OpenAI. Fields: `tenant_id`, `client_id`, `client_secret`, `access_token`, `refresh_token`, `expires_at`. Added to `CredentialUnion`.

### models.dev integration

- **`models_dev.py` restored** — live fetch from `https://models.dev/api.json`, on-disk cache at `~/.local/share/authlm/models-dev-cache.json`, vendored fallback snapshot in the wheel.
- **`authlm refresh-registry` command** — explicit refresh of the models.dev cache. Fetches, validates, writes to cache. Soft-fails on network error (existing cache untouched, non-zero exit).
- **Providers source metadata from models.dev** — `display_name`, `docs_url`, `logo_url` come from `get_provider_metadata(provider_id)` instead of hardcoded values. Auth-specific config (OAuth endpoints, client IDs, scopes) stays in `_auth_table.py`.
- **`scripts/refresh_models_dev.py` restored** — operator script to re-vendor the snapshot JSON. Run manually before releases.

### CLI additions

- **`authlm import llm`** — one-way migration from `llm`'s plaintext key store (`~/.config/io.datasette.llm/keys.json`). For each `(provider, key)` pair, stores via `authlm.connect()`. Does not delete the `llm` key file.
- **`authlm export [--format json|env] [--output <file>]`** — portable plaintext export of all stored credentials. Users are warned that the export file is plaintext. For migration and disaster recovery.
- **Interactive CLI menu** — richer method selection (fuzzy search, arrow keys) via `rich` or `questionary`. Replaces v0.1.0's numbered-list prompt.

### Documentation & distribution

- **Doc site** — `docs/` directory with `index.md`, `quickstart.md`, `plugin-authoring.md`, `security-model.md`, `api/`. Published to `authlm.readthedocs.io`. `docs.yml` GitHub Actions workflow.
- **`examples/` directory** — runnable examples: `connect_api_key.py`, `connect_oauth.py`, `fastapi_integration.py`, `agent_sdk.py`.
- **Homebrew tap** — `brew install authlm` on macOS. Separate tap repository (`homebrew-authlm`).
- **`CONTRIBUTING.md` "How to add a provider" section** — expanded with plugin authoring guide for v0.2.0.

### Security

- **`Field(repr=False)` on all secret fields** — carried from v0.1.0. New credential types (`AwsCredential`, `AzureAdCredential`) follow the same pattern.
- **`redact_body` applied to all OAuth error sites** — carried from v0.1.0. New providers' token endpoints use the same redaction.

## Deferred from v0.2.0

- **File-locking / multi-process refresh safety** — deferred to v0.3.0 (robustness theme). v0.2.0 remains single-process.
- **Additional store backends (Vault, Bitwarden, 1Password)** — deferred to v0.3.0.
- **Audit log** — deferred to v0.3.0.
- **Adapters as maintained API** — deferred to v0.3.0 (gated on ≥3 independent user requests).
- **`client_credentials` OAuth grant** — deferred to v0.3.0.

---

## Known limitations (carry from v0.1.0)

- **Single-process assumption** — multi-process apps refreshing the same credential concurrently may invalidate each other's refresh tokens. File-locking lands in v0.3.0.
- **No proxy/middleware support** — AuthLM does not inject credentials into API calls. Use LiteLLM or a local proxy with AuthLM as the credential source.
- **Linux keychain variance** — headless Linux without Secret Service falls back to `EncryptedFileStore`.
