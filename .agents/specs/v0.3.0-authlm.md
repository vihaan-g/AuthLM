# AuthLM v0.3.0 — Robustness & More Stores

**Version:** 0.3.0
**Status:** Planned (outline)
**SemVer:** Pre-release — API may break. Plugin hookspec signatures stabilize here; breaking changes after v0.3.0 require a major version bump within 0.x.
**Prerequisite:** v0.2.0 shipped.

---

## Theme

AuthLM is safe to use in production at scale. Enterprise secret managers are supported via plugin stores. An audit trail records credential access. The concurrency limitation from v0.1.0 is resolved with file-locking. Maintained SDK adapters are promoted from docs examples to real API.

---

## Scope

### Concurrency

- **File-locking for multi-process refresh safety** — `fcntl.flock` (POSIX) / `msvcrt.locking` (Windows) around `CredentialStore.set()` and `refresh()`. Two processes refreshing the same credential simultaneously no longer invalidate each other's refresh tokens. Resolves the v0.1.0 §17.2 known limitation.
- **Lock granularity** — per `(provider, alias)` pair, not global. Lock file at `~/.local/share/authlm/locks/<provider>:<alias>.lock`.
- **Timeout** — configurable lock timeout (`AUTHLM_LOCK_TIMEOUT_SECONDS`, default 10s). On timeout, raises `AuthLMError("credential lock timeout; another process may be refreshing"`).

### Additional store backends

- **HashiCorp Vault** — `authlm-vault` plugin package. Reads/writes credentials via the Vault HTTP API. Supports KV Secrets Engine v2. `AUTHLM_STORE=vault` or `authlm connect openai --store vault`.
- **Bitwarden** — `authlm-bitwarden` plugin package. Reads credentials from the Bitwarden CLI (`bw`) vault. Read-only.
- **1Password** — `authlm-1password` plugin package. Reads credentials from the 1Password CLI (`op`) vault. Read-only.
- **AWS Secrets Manager** — `authlm-aws-secrets` plugin package. Reads/writes via boto3. `boto3` is an optional dependency of the plugin, not of core authlm.
- **GCP Secret Manager** — `authlm-gcp-secrets` plugin package. Reads/writes via google-cloud-secret-manager.

Each plugin is a separate PyPI package with its own repo, tests, and release cycle. The AuthLM core team maintains the Vault plugin; others are community-maintained.

### Audit log

- **Append-only audit log** — every `get_credential`, `get_valid_credential`, `refresh`, `validate`, `connect`, `disconnect` appends a JSON line to the audit log.
- **Location** — `~/.local/share/authlm/audit.log` (configurable via `AUTHLM_AUDIT_PATH`).
- **Schema** — `{"timestamp": "...", "action": "get_credential", "provider": "openai", "alias": "default", "result": "success"}`. No secrets in the log.
- **CLI** — `authlm audit [--provider <id>] [--since <date>]` prints the audit log with filters.
- **Rotation** — log rotates at 10MB (configurable). Old logs kept for 90 days.

### Adapters (maintained API)

- **`as_openai_client(cred)`** — returns an `openai.OpenAI` client configured with the credential's API key or access token. Promoted from docs example to maintained API with full test coverage and SemVer guarantees.
- **`as_anthropic_client(cred)`** — returns an `anthropic.Anthropic` client.
- **`as_google_client(cred)`** — returns a `google.genai.Client`.
- **Gating criterion** — per v0.1.0 §17.1: at least 3 independent user requests for the same adapter. If demand is not met by v0.3.0, adapters stay as docs examples and are re-evaluated for v0.4.0.

### Additional OAuth grants

- **`client_credentials` grant** — `OAuthClientCredentialsMethod` for server-to-server auth (no user involvement). Used by providers that issue machine-to-machine tokens (e.g., Azure OpenAI with AAD app registration).
- **New `OAuthGrant` enum value** — `CLIENT_CREDENTIALS = "client_credentials"`.
- **`resource_owner_password` grant** — not planned. Legacy and deprecated by OAuth 2.1. If a provider requires it, consumers can implement a custom `ConnectionMethod` via the plugin system.

## Deferred from v0.3.0

- **TUI mode** — a Textual-based TUI for the `connect` flow is a candidate but not planned. Depends on whether the v0.2.0 interactive CLI menu proves painful enough to warrant a richer UI. Re-evaluated after v0.2.0 adoption.
- **TypeScript SDK** — a companion `@authlm/node` package is not planned for v0.3.0. If there's demand from JS AI apps, it's a separate project, not an AuthLM version.

---

## Known limitations (carry from v0.2.0)

- **No proxy/middleware support** — AuthLM does not inject credentials into API calls. Use LiteLLM or a local proxy with AuthLM as the credential source.
- **Linux keychain variance** — headless Linux without Secret Service falls back to `EncryptedFileStore`.
- **Claude Pro/Max OAuth is shipped but warned** — Anthropic prohibits this in their ToS. Integrators are responsible for compliance.
